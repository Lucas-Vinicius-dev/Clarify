Tier B — Boas, opcionais Schema-driven UI

Formulário dinâmico
por tipo de demanda.

Cada tipo de demanda acadêmica precisa de informações diferentes — justificativa de falta pede atestado e data, aproveitamento de créditos pede ementa e carga horária. Em vez de um textarea genérico, um schema declarativo descreve os campos exigidos e o modal monta o formulário certo na hora.

Schema JSON Conditional ServiceNow-like

Por que isso importa

Hoje, o modal de nova demanda em Clarify/src/components/structures/modais.js (linhas 74–158) pede só dois campos: título e descrição com mínimo de 10 caracteres. O coordenador recebe um parágrafo livre — e quase sempre precisa pedir os dados que faltam. Isso vira o status requer_ajuste, três mensagens a mais e dias de atraso. Um Google Form mal-feito tem mais estrutura que isso.

01

Descrição genérica gera retrabalho

Em templateNovaDemanda existe um único <textarea> de descrição. Para uma justificativa de falta, o aluno esquece a data; para aproveitamento, esquece a carga horária. Coordenador devolve para complementar.

02

Vista de revisão é um blob

templateDetalhes (linha 335) joga um whitespace-pre-wrap com tudo dentro. Coordenador tem que ler um parágrafo livre para extrair "qual disciplina? qual data? cadê o atestado?".

03

Tipo de demanda é apenas um título

Hoje o "tipo" é só o conteúdo do campo titulo que o aluno escreve. Não existe taxonomia, não existe contrato de quais dados o tipo X exige. É texto livre disfarçado de categoria.

Como funcionaria — fluxo

Do clique em "Nova solicitação" até o coordenador ver dados estruturados, com schema versionado por trás.

  1. Aluno escolhe o tipo no topo do modal

    O atual campo único de título vira um seletor visual (radios) com os 5 tipos canônicos. Cada tipo carrega o schema correspondente de formSchemas.js.

  2. Renderer monta o form a partir do schema

    Um novo módulo formRenderer.js recebe a árvore JSON de campos e devolve HTML — substituindo o bloco fixo do templateNovaDemanda. Cada campo carrega seu tipo (select / date / textarea / anexo / radio / number), label, help text e regras.

  3. Campos condicionais aparecem conforme o aluno responde

    Trancamento de matrícula com motivo "saúde" revela campo "atestado médico" obrigatório. Regras de visibilidade são parte do schema (visivel_se: {{ "motivo": "saude" }}).

  4. Validação acontece antes de habilitar "Enviar"

    Substitui a regra atual titulo >= 4 && descricao >= 10 por validação por campo (obrigatórios preenchidos, min_len, max em datas, tipos aceitos em anexos).

  5. Demanda persiste com tipo_id + schema_versao + campos_preenchidos

    O modelo atual ganha duas chaves novas. descricao permanece como campo opcional dentro do schema, para casos que ainda precisam de texto livre.

  6. Coordenador vê os mesmos campos estruturados na revisão

    O bloco "Detalhes da solicitação" em templateDetalhes (linha 329) é substituído por uma renderização baseada no schema da época em que a demanda foi criada — mesmo que a instituição atualize depois.

Inspiração de mercado

Schema-driven forms são padrão fora do ensino: Zendesk faz isso com "ticket forms", ServiceNow gira em torno disso, e o ecossistema JSON Schema tem uma referência aberta.

Referência O que faz Adaptação para Clarify
Zendesk Ticket Forms
desk365.io		 Cada tipo de ticket tem campos próprios; campos podem ser mostrados condicionalmente. Branding por tipo na customer-facing form. 5 tipos canônicos de demanda acadêmica + schema condicional (saúde → atestado). Mesma ideia, vocabulário acadêmico.
ServiceNow Forms
platform-default		 UI inteiramente schema-driven. Form designer drag-and-drop. Versionamento de catálogos de serviço. Pegamos só o essencial: editor visual simples + versionamento. Sem o overhead enterprise.
JSON Schema + react-jsonschema-form
json-schema.org		 Padrão aberto para validar JSON; bibliotecas que renderizam form a partir de schema. Inspiração técnica para o formRenderer.js. Não usamos a lib (peso desnecessário), mas seguimos o vocabulário (type, required, minLength).
Typeform / Tally
typeform.com		 Form em passos com indicador "3 de 7". Lógica condicional (jump logic). Indicador de progresso "4/6 campos" no header do modal. Reduz ansiedade de "quando isso acaba?".

Métricas de sucesso

Cinco indicadores que validariam a feature. O primeiro é o coração: se demanda chega completa, todo o resto melhora.

+50%
Demandas completas na 1ª tentativa

% de demandas que não precisam voltar com requer_ajuste só por dado ausente. Baseline: medir hoje.

−40%
Mensagens "preciso que você complete..."

Threads iniciadas pelo coordenador pedindo dado faltante. Tracking por classificação simples no audit log.

−2d
Tempo médio de resolução

Por tipo. Cai porque coordenador para de esperar o aluno completar a demanda — decide de uma vez.

≤ +30s
Tempo extra do aluno preenchendo

Não dá pra cansar o aluno. Form precisa demorar pouco mais que o genérico atual, ou desincentiva a usar.

100%
Cobertura de schema por tipo

Demandas com tipo_id identificado e schema correspondente. Versão completa = todos os 5 tipos canônicos cobertos.

Quick wins vs. versão completa

MVP entrega valor real para 2 dos 5 tipos sem editor visual. Versão completa institucionaliza o conceito.

MVP · 1–2 dias apresentação

  • Schema hardcoded em formSchemas.js para 2 tipos (Justificativa de Falta + Aproveitamento de Créditos)
  • formRenderer.js com suporte a text, textarea, select, date, anexo (sem condicional)
  • Modal nova demanda renderiza form a partir do schema escolhido
  • Validação por campo (obrigatórios e min_len)
  • Vista do coord exibe campos estruturados em vez de blob

Versão completa produto real

  • 5 tipos canônicos cobertos + "Outra" como escape
  • Campos condicionais (visivel_se) e validação cruzada
  • Editor visual drag-and-drop para coordenação de curso criar/editar schemas
  • Versionamento real: valida_de, snapshot por demanda, migração transparente
  • Override por curso (curso de Enfermagem pode pedir campo extra em Aproveitamento)
  • Auto-save de rascunho a cada 5s no localStorage

Riscos & armadilhas

Schema-driven UI tem armadilhas conhecidas. Lista os quatro mais prováveis neste contexto.

Form cansa o aluno UX · alto
Form de aproveitamento tem 6 campos vs. 2 atuais — pode parecer trabalho. Mitigação: indicador "4/6 campos" no header, salvar rascunho a cada 5s, campos opcionais visualmente diferentes, help_text em linguagem acessível ("em qual data você faltou?" em vez de "informe a data de ocorrência da ausência").
Schema não cobre caso real Cobertura · médio
Aluno tem situação atípica que não cabe em nenhum tipo. Mitigação: tipo "Outra solicitação" com schema mínimo (título + descrição livre) + campo "Observações adicionais" opcional em todos os tipos como escape sancionado.
Manter schemas atualizados vira trabalho Operacional · médio
Tipos de demanda são bem padronizados no contexto acadêmico (LDB define a maioria), então o catálogo é estável. Mitigação: versionamento por schema permite mudanças sem migração massiva. Demandas antigas continuam mostrando o schema da época.
Migração das demandas existentes Backwards-compat · baixo
Demandas antigas têm só tipo (texto livre) e descricao. Mitigação: adicionar schema_versao: 0 como "schema legado" no renderer — renderiza como texto livre. Sem migração destrutiva. Demandas novas usam o schema novo, antigas continuam exibindo igual.
Editor visual vira complexo demais Scope creep · alto
Drag-and-drop com lógica condicional é um produto inteiro (vide Typeform). Mitigação: MVP só edita schemas em JSON via arquivo. Editor visual é V2 e pode ficar restrito a coord-master (não cada coord criando novos tipos).
