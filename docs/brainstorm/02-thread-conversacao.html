Tier S — Mudam o jogo Conversa, não formulário

A demanda vira conversa real,
não um campo de feedback único.

Hoje uma demanda tem feedback: "string" — uma frase só do coord ao reprovar. É um Google Form com status. A Feature 02 transforma cada ticket em thread completa: mensagens datadas entre aluno e coord, notas internas visíveis só ao time, mentions, indicador de digitação e detecção de colisão estilo Help Scout.

Thread Notas internas Help Scout Zendesk Public/Internal Mudança estrutural

Por que isso importa

A Carla mandou atestado, o Bruno pediu "anexe novamente", a demanda virou requer_ajuste com uma string de 60 caracteres — e acabou. Se a Carla quer perguntar "qual foi o problema do atestado anterior?", não tem onde. Ela abre uma nova demanda. O processo trava no que deveria ser uma troca de duas mensagens.

O coord só pode "responder uma vez". Em dashboardcoord.js:49, reprovarDemanda() abre um prompt e grava demanda.feedback = feedback. Não há append, não há histórico. Reprovar duas vezes sobrescreve a observação anterior.
O aluno só lê, nunca responde. centralDemandas.js:82 renderiza ${demanda.feedback} como bloco estático. Sem botão de réplica, sem campo de digitação. A demanda é write-only para o coord.
Não existe espaço privado para discussão. Quando o caso da Carla é dúvido (atestado de pronto-socorro estranho, prazo de 5 dias), o Bruno precisa perguntar à Ana antes de responder. Hoje isso acontece no WhatsApp e some do registro institucional.
O modal de detalhes (modais.js:375-383) tem uma única "Observação da coordenação". O design já reserva o espaço — falta torná-lo um feed cronológico em vez de um break-words whitespace-pre-wrap de texto único.

Como funcionaria — fluxo

Cenário: Carla abre uma justificativa de falta. Bruno acha o atestado ilegível. Ele consulta Ana antes de responder. Fluxo ponta a ponta.

Carla abre a demanda com mensagem

Em vez de descricao: "string" isolada, a abertura já é a primeira mensagem da thread. Anexa o atestado direto na mensagem. Status entra como pendente.

Bruno abre o ticket e começa a digitar uma nota interna

Toggle no campo de composição vira "Nota interna" (fundo amarelo). Ele escreve "@ana.silva esse atestado tá legível pra você?". Ana recebe notificação.

Ana responde também em nota interna

"O carimbo está borrado, peça reenvio." A Carla nunca vê essa troca — o campo tipo: "nota_interna" filtra na renderização. Cada mensagem tem timestamp.

Bruno troca para mensagem pública e responde

Toggle volta a "Mensagem pública" (fundo branco/laranja). Status muda automático para requer_ajuste, mas agora é uma mensagem datada na timeline, não um blob estático em demanda.feedback.

Collision detection avisa quando alguém mais digita

Se a Ana resolve responder pública (escalou pra coord. pedagógico), o Bruno vê um banner amarelo: "Ana Silva está respondendo agora." Evita resposta duplicada e contraditória — padrão do Help Scout.

Carla reenvia e a thread fecha

Ela vê "Bruno digitando…", recebe a resposta com indicador "lido às 14:32", anexa o atestado novo. Status volta a em_analise. Tudo na mesma demanda, sem protocolo novo.

Modelo de dados

O schema atual está em Clarify/src/data/demanda.js e na criação implícita em dashboardcoord.js:49-72. A mudança é trocar o campo único feedback por um array mensagens[].

Schema atual

{
  protocolo: "#A1F3-2026",
  tipo: "Justificativa de Falta",
  status: "requer_ajuste",
  descricao: "Faltei na prova...",
  matriculaAluno: "20240003",
  dataCriacao: "2026-05-03",
  dataAtualizacao: "2026-05-03",
  feedback: "Anexe o atestado novamente"
  // ↑ string única.
  //   sobrescrita a cada reprovação.
}

Schema proposto

{
  protocolo: "#A1F3-2026",
  tipo: "Justificativa de Falta",
  status: "requer_ajuste",
  matriculaAluno: "20240003",
  dataCriacao: "2026-05-03",
  dataAtualizacao: "2026-05-03",
  participantes: [
    "20240003", // Carla
    "20240002", // Bruno
    "20240001"  // Ana (observadora)
  ],
  mensagens: [
    {
      id: "msg_001",
      autor_matricula: "20240003",
      autor_nome: "Carla Pereira",
      autor_tipo: "aluno",
      tipo: "publica",
      texto: "Faltei na prova...",
      timestamp: "2026-05-03T10:30:00Z",
      lida_em: "2026-05-03T11:45:00Z",
      anexos: ["arq_atestado_v1"],
      mentions: []
    },
    {
      id: "msg_002",
      autor_matricula: "20240002",
      autor_nome: "Bruno Almeida",
      autor_tipo: "coordenador",
      tipo: "nota_interna",
      texto: "@ana.silva carimbo ilegível?",
      timestamp: "2026-05-03T14:00:00Z",
      lida_em: null,
      anexos: [],
      mentions: ["20240001"]
    }
  ]
}

Mudança não-destrutiva: para demandas legadas, um migrator simples converte feedback: "..." em mensagens: [{ tipo: "publica", autor_tipo: "coordenador", texto: feedback, ... }]. Sem perda de dado.

Inspiração de mercado

Help desks resolveram este problema há 15 anos. A tarefa não é inventar — é adaptar ao contexto acadêmico.

Referência O que fazem Adaptação para o Clarify
Help Scout
Shared inbox 		Conversas por e-mail viram threads compartilhadas. Notas internas amarelas. Mentions com @. Collision detection: vê quem está digitando antes de você apertar enviar. Coord. de curso e coord. pedagógico operam na mesma "caixa" de demandas. Mention escala caso difícil. Indicador de colisão evita resposta dupla quando dois coords abrem o mesmo ticket.
Zendesk
Agent workspace 		Botões físicos separados: Public reply e Internal note. O agent precisa escolher conscientemente — fundo do composer muda de cor. Macros e KB embutidos. Toggle binário "Pública / Interna" no composer com mudança visual forte (fundo amarelo na nota). Reduz o risco de vazar uma nota interna pro aluno por engano.
Freshdesk
Internal threads 		Conversa privada paralela à pública. Múltiplos agents discutem sem cliente ver. Trilha de discussão sobrevive ao caso encerrado. Histórico de discussão fica institucional. Quando a Carla pergunta no semestre seguinte "por que vocês negaram em 2026?", a thread interna está lá, datada, auditável.
Intercom Inbox
Hybrid view 		Mostra mensagens públicas e internas na mesma timeline, com tratamento visual distinto. Receipts ("seen 2 min ago"), typing dots, presence indicator. Vista unificada do coord: nota e mensagem pública na mesma rolagem. Receipts para o aluno saber se a coordenação leu. Typing dots quando o coord está formulando resposta.

Métricas de sucesso (KPIs)

Como saber se a feature está pagando o esforço. Cinco KPIs concretos, mensuráveis em cima do schema novo.

> 3 Mensagens / demanda (média) Hoje é ≈ 1 (descrição + feedback opcional). Subir pra 3 indica que a thread virou ferramenta real, não decoração.
↓ 50% Demandas duplicadas pelo mesmo aluno Aluno que não consegue responder no ticket abre um novo. Thread real elimina esse padrão. Medir por matriculaAluno + tipo em janela de 30 dias.
FRT First Response Time mediano Tempo entre dataCriacao e timestamp da primeira mensagem pública do coord. Alvo inicial: < 24h em dias úteis.
15–25% % de demandas com nota interna Proxy de complexidade institucional. Acima de 40% indica que casos simples estão sendo "discutidos demais" — gatilho pra revisar política.
≥ 90% Read rate das respostas do coord Quantas mensagens públicas têm lida_em != null em até 48h. Mede se o aluno está realmente acompanhando.

Quick wins vs. versão completa

O MVP cabe em 1–2 dias de trabalho e já é apresentável. A versão completa é o objetivo de médio prazo.

MVP — 1 a 2 dias

  • Array de mensagens com tipo público/interno e timestamp. Sem typing, sem read receipt, sem collision.
  • Render cronológico no modal de detalhes — substitui o bloco único de "Observação".
  • Composer com toggle Pública/Interna. Mudança de cor de fundo é o feedback visual.
  • Migrator para demandas legadas (feedback → primeira mensagem da thread).
  • Filtro de renderização: aluno só recebe tipo === "publica".

Versão completa

  • Mentions com autocomplete de data/usuarios.js e notificação visível ao mencionado.
  • Read receipts (lida_em) com ícone de check duplo, estilo Intercom.
  • Typing indicator e collision detection via polling em localStorage (presença simulada).
  • Anexos por mensagem (integra Feature 07) — Carla pode reenviar o atestado dentro da resposta.
  • Professor como observador: campo participantes[] permite que Ana veja a thread dos seus orientados.
  • Export auditável: ao concluir, gerar PDF da thread filtrada (sem notas internas) para o aluno.

Riscos & armadilhas

Cinco perigos concretos. Mais grave: vazamento de nota interna. Menos grave mas mais provável: estourar o limite do localStorage.

Nota interna vazando para o aluno

Se a renderização do modal do aluno (centralDemandas.js) usa o mesmo template do coord, basta um if esquecido para que o aluno veja "@ana.silva esse caso é boboca". Catástrofe institucional.

Mitigação: filtrar tipo === "publica" na camada de dados, não na de UI. Mesmo que o template renderize tudo, o array que chega ao aluno já vem peneirado. Adicionar testes que negam acesso à nota_interna a partir de autor_tipo: "aluno". Em exports/PDF, mesma lógica.

Volume estourando o localStorage

Limite típico é 5–10 MB por origem. Com 500 demandas × 5 mensagens × 800 chars + anexos referenciados, dá pra encostar no teto rápido. App quebra silencioso.

Mitigação: paginar mensagens (carregar últimas 20, "ver anteriores" sob demanda) e mover demandas status: "concluido" com mais de 60 dias para um bucket de "arquivadas" comprimidas. Como é TCC sem backend, isso é workaround consciente — documentar.

Spam de mentions

Aluno descobre que pode escrever @bruno.almeida e marcar o coord a cada 5 minutos. Coord vira notification-zombie e desliga tudo.

Mitigação: aluno só pode mencionar coordenadores atribuídos à sua demanda (ou orientador via participantes). Coordenadores podem mencionar qualquer um. Rate-limit local: ignora mention repetido na mesma thread em < 30 min.

Estética virando WhatsApp e perdendo seriedade

Bolinhas verdes, "✓✓ azul", emojis de carinha — UI passa a parecer caixinha de bate-papo e mina a percepção de processo institucional. Coordenador para de levar a sério.

Mitigação: manter paleta Clarify (laranja queimado, slate, surface creme). Timestamps absolutos ("14:32 · 03 mai") e não relativos ("há 2 min"). Sem emojis de reação. Status oficial da demanda continua dominante no header do modal — não na conversa.

Status "requer_ajuste" dessincronizando da última mensagem

Coord escreve "anexe novamente" mas esquece de mudar o status. Aluno vê em_analise, não sabe que precisa agir. Pior do que hoje.

Mitigação: ao enviar mensagem pública, oferecer dropdown opcional "Marcar como…" (requer_ajuste / em_analise / concluido). Default empilhado em cima do botão Enviar — coord vê e escolhe consciente, não automático.

Resumo executivo

Trocar feedback: "string" por mensagens[] é a mudança mais barata e mais transformadora deste roadmap. Não exige IA, não exige backend, não exige nova tela — só refatorar o que já está em modais.js e dashboardcoord.js. Em troca, o Clarify deixa de ser ticket-tracker e passa a ser um help desk de verdade, com colaboração interna, histórico auditável e o aluno na ponta do diálogo, não na ponta de uma string.
