Fundamental EDU 01

Estado React: 3 camadas

O Clarify usa quatro sistemas de estado independentes que coexistem sem conflito. Cada um resolve um problema específico. Não há "estado global" — há estado de servidor, estado de UI, estado de tema e estado de autenticação, cada um no seu container.

React Context Zustand TanStack Query
Visão geral

Três camadas, três responsabilidades

Não há sobreposição: cada camada cuida de um tipo de estado e não interfere nas outras. A regra é: estado de servidor vai pro React Query, estado de UI vai pro Zustand, estado de auth vai pro Context.

Camada 1

Dois contextos, duas responsabilidades

Diferente de apps que criam dezenas de contextos, o Clarify tem apenas dois: ThemeContext para tema claro/escuro e AuthContext para autenticação. Cada um existe porque ambos precisam ser globais: qualquer página ou componente pode chamar useTema() ou useAuth() sem prop drilling.

Camada 2

Zustand — estado de UI sem boilerplate

O Zustand substitui Redux sem contexto, sem provider, sem dispatch. Duas stores pequenas e focadas, cada uma com responsabilidade única. Não usamos Context + useReducer porque Zustand dá menos voltas para o mesmo resultado.

Camada 3

TanStack React Query — cache de servidor

Toda comunicação com o backend passa pelo React Query. Ele gerencia cache, loading states, erros e revalidação. Não usamos useEffect para buscar dados — só React Query.

Hierarquia

Como os providers se aninham

A ordem importa: QueryProvider precisa vir antes de qualquer hook React Query, AuthProvider precisa vir antes de qualquer useAuth().

src/app/layout.tsx (Server Component — único sem 'use client')
ThemeProvider <ThemeContext.Provider> com useState lazy + localStorage src/context/ThemeContext.tsx
↓
QueryProvider <QueryClientProvider> com client criado via useState lazy src/components/providers/QueryProvider.tsx
↓
AuthProvider <AuthContext.Provider> com useState + useEffect + onAuthStateChange src/context/AuthContext.tsx
↓
{children} Páginas: /login, /registro, /(dashboard)/*, /perfil/*
Ordem dos providers: ThemeProvider > QueryProvider > AuthProvider. O ThemeProvider é o mais externo porque o tema precisa estar disponível em toda a app, incluindo o próprio QueryProvider e AuthProvider. O QueryProvider vem antes do AuthProvider por convenção — os hooks de dados (useDemandas, useTurmas...) são chamados dentro das páginas, que estão dentro de todos os providers.
