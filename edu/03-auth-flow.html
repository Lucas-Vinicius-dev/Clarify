Fundamental EDU 03

Autenticação: Cookie + Proxy + Context

O Clarify usa um modelo híbrido: o AuthContext gerencia a sessão no cliente, o proxy.ts faz o gating server-side lendo um cookie, e as API routes autenticam via Supabase SSR. Duas camadas de proteção, zero flash de conteúdo.

Cookie clarifysessao: Gravado por defineCookieSessao(cargo) (AuthContext.tsx:26) e lido por request.cookies.get('clarify_sessao') no proxy.ts:13. O cookie contém apenas o cargo ('aluno' ou 'coordenador') — nunca tokens ou senhas. Expira em 7 dias. O proxy rejeita qualquer valor fora desses dois, evitando cookie adulterado.
Máquina de estados

Estados do AuthContext

O AuthContext tem 3 estados possíveis, e cada página reage de forma diferente a cada um.

loading = true Carregando Supabase auth.getSession() em andamento
→ resolve →
usuario != null Autenticado Cookie definido. Proxy permite acesso à rota do cargo.
usuario = null, loading = false Não autenticado Proxy redireciona para /login. Páginas mostram formulário. 
// src/context/AuthContext.tsx — Ciclo de vida da sessão

// 1. Ao montar: verifica sessão existente
useEffect(() => {
  supabase.auth.getSession().then(({ data: { session } }) => {
    if (session?.user) fetchProfile(session.user.id)  // loading = true até resolver
    else setAuthState(prev => ({ ...prev, loading: false })) // sem sessão
  })

  // 2. Escuta mudanças de auth em tempo real
  const { data: { subscription } } = supabase.auth.onAuthStateChange(
    (_event, session) => {
      if (session?.user) fetchProfile(session.user.id)
      else setAuthState({ usuario: null, loading: false })
    }
  )

  return () => subscription.unsubscribe() // cleanup
}, [supabase, fetchProfile])
Proxy

Server-side gating com Next.js 16 proxy

O antigo "middleware" no Next.js 16 chama-se proxy. Ele roda antes de qualquer página ser servida e toma decisões de rota baseadas no cookie de sessão.

Lógica do proxy

src/proxy.ts
export function proxy(request) {
  const cargo = request.cookies.get('clarify_sessao')?.value

  // 1. Sem cookie válido → /login
  if (cargo !== 'aluno' && cargo !== 'coordenador')
    return redirect('/login')

  // 2. Coord tentando /centraldemandas → redireciona
  if (pathname.startsWith('/centraldemandas') && cargo !== 'aluno')
    return redirect('/dashboardcoord')

  // 3. Aluno tentando /dashboardcoord → redireciona
  if (pathname.startsWith('/dashboardcoord') && cargo !== 'coordenador')
    return redirect('/centraldemandas')

  return NextResponse.next()
}

Rotas protegidas

O matcher do proxy só dispara para:

←
/dashboardcoord/:path*
Coordenação
←
/centraldemandas/:path*

Páginas públicas (/login, /registro) não passam pelo proxy. O login ja estando autenticado redireciona via useRouter no cliente.
Registro

Como um novo usuário é criado

O registro é o único fluxo que usa o cliente admin do Supabase, porque precisa criar um usuário no Auth sem que ele esteja logado.

Form RegistroPage useForm + zodResolver
→
useAuth().registro() Valida chave + cria
→
POST /api/auth/register Admin client
→
Ações 1. Validar chave
2. admin.createUser()
3. Marcar chave usada
Segurança: A chave de ativação é validada pela service-role key (admin client). Isso impede que um aluno tente criar várias contas. O registro só cria coordenadores — alunos são adicionados pelo coordenador após o login.
